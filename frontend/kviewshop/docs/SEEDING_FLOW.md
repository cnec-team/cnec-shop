# 제품 체험 신청 (시딩 플로우) — 서비스 전략

## 1. 개요

크넥샵의 3번째 커머스 축.
- 공구 (기간한정 특별가) — 트래픽 폭발
- 크리에이터픽 (상시 큐레이션) — 샵 정체성
- 제품 체험 신청 (시딩) — 신뢰 기반 판매의 시작점

기존 크넥 에이전시에서 수동으로 하던 시딩(카톡/DM으로 "이거 써볼래요?")을 플랫폼에 내재화.
대기 고객사 1,000개+ 규모에서 수동 시딩은 물리적 한계 → 시스템이 필수.

## 2. 4자 관점 가치

### 크리에이터
- "써보지도 않은 제품을 어떻게 추천해?" → 체험 후 확신 있게 추천
- 의무 없음. 안 맞으면 패스. 불이익 없음
- 체험→공구 전환 시 "직접 써봤어요" = 팔로워 신뢰 극대화
- 새 브랜드 발견 채널 — 크리에이터가 먼저 손 뻗을 수 있음

### 브랜드
- 승인권 100% 보유 → 원하는 크리에이터에게만 샘플 발송
- 무작위 시딩이 아닌 타겟 시딩 → 샘플 비용 효율화
- 체험 후 공구 전환율 데이터 = 크리에이터 선별 기준
- 크리에이터가 먼저 관심을 보인다 = 진정성 있는 파트너

### 크넥(플랫폼)
- 체험→공구 전환율 데이터 축적 → AI 매칭 고도화의 핵심 입력값
- "크넥에서 신청하면 브랜드가 보내줌" = 크리에이터가 크넥을 쓰는 이유 추가
- 시딩 볼륨 데이터 = 브랜드에게 보여줄 성과 지표
- 에이전시 시딩 운영 부담 경감

### 구매자(팔로워)
- "이 크리에이터가 직접 써봤구나" → 구매 전환율 상승
- 체험 기반 추천 vs 광고성 추천의 신뢰 차이

## 3. 상태 머신

```
REQUESTED ──→ APPROVED ──→ SHIPPED ──→ RECEIVED ──→ DECIDED
    │              │                                    ├─→ PROCEED (공구/픽 전환)
    │              │                                    └─→ PASS (종료, 불이익 없음)
    │              └──→ REJECTED (종료)
    └──→ CANCELLED (크리에이터가 취소)
```

### 상태별 설명

| 상태 | 주체 | 설명 |
|------|------|------|
| REQUESTED | 크리에이터 | 상품 카탈로그에서 "체험 신청" 클릭 |
| APPROVED | 브랜드 | 크리에이터 프로필 확인 후 승인 |
| REJECTED | 브랜드 | 거절 (사유 선택: 카테고리 불일치 / 팔로워 기준 미달 / 재고 부족 / 기타) |
| SHIPPED | 브랜드 | 샘플 발송 완료 (송장번호 입력) |
| RECEIVED | 크리에이터 | 수령 확인 (자동: 배송완료 후 3일 / 수동: 크리에이터 확인) |
| DECIDED - PROCEED | 크리에이터 | 공구 캠페인 참여 또는 크리에이터픽 추가 |
| DECIDED - PASS | 크리에이터 | 패스. 사유 선택 (선택사항): 제품 안 맞음 / 팔로워 반응 예상 안 좋음 / 기타 |
| CANCELLED | 크리에이터 | 승인 전 취소 |

## 4. 데이터 모델 (향후 구현)

### ProductTrial 테이블

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| creator_id | UUID | FK → Creator |
| product_id | UUID | FK → Product |
| brand_id | UUID | FK → Brand |
| status | ENUM | REQUESTED / APPROVED / REJECTED / SHIPPED / RECEIVED / DECIDED / CANCELLED |
| decision | ENUM | NULL / PROCEED / PASS |
| reject_reason | TEXT | 거절 사유 (nullable) |
| pass_reason | TEXT | 패스 사유 (nullable) |
| tracking_number | VARCHAR | 송장번호 (nullable) |
| requested_at | TIMESTAMP | 신청 시각 |
| responded_at | TIMESTAMP | 승인/거절 시각 |
| shipped_at | TIMESTAMP | 발송 시각 |
| received_at | TIMESTAMP | 수령 시각 |
| decided_at | TIMESTAMP | 결정 시각 |
| converted_to | VARCHAR | 전환 대상 (campaign_id 또는 shop_item_id, nullable) |
| created_at | TIMESTAMP | |
| updated_at | TIMESTAMP | |

### 인덱스

- `(creator_id, status)` — 크리에이터별 체험 현황 조회
- `(brand_id, status)` — 브랜드별 신청 관리
- `(product_id)` — 상품별 체험 통계

## 5. UI 위치

### 크리에이터 센터 (creator.cnec.kr)

- **상품 카탈로그**: 각 상품 카드에 "체험 신청" 버튼 (`allow_trial = true`인 상품만)
  - 이미 신청한 상품: "체험 진행중" 상태 표시
  - 이미 공구/픽 전환한 상품: "체험 완료" 표시
- **내 체험 현황**: 신청 목록 + 상태 타임라인
  - 상태별 필터: 전체 / 대기중 / 진행중 / 완료
- **체험 완료 후 CTA**:
  - "이 제품으로 공구 참여하기" → 진행중 캠페인 연결
  - "크리에이터픽에 추가하기" → 1탭 추가

### 브랜드 어드민 (brand.cnec.kr)

- **체험 신청 관리**: 신청 목록 + 크리에이터 프로필 확인
  - 크리에이터 정보: 팔로워 수, 피부타입, 카테고리, 클린스코어
  - 승인/거절 버튼 + 거절 사유 선택
  - 승인 후 송장번호 입력
- **상품 설정**: "체험 신청 허용" 토글 (`allow_trial`, 기본값 ON)
- **체험→공구 전환 리포트**:
  - 어떤 크리에이터가 체험 후 실제 공구했는지
  - 체험→공구 전환율
  - 체험 후 패스한 크리에이터의 사유 분포

### 크넥 어드민 (admin.cnec.kr)

- 전체 체험 신청 모니터링
- 체험→공구 전환율 대시보드
- 브랜드별 시딩 현황 + 평균 응답 시간

## 6. 기존 플로우와의 관계

| 플로우 | 시작 주체 | 체험 여부 | 관계 |
|--------|----------|----------|------|
| 공구 캠페인 | 브랜드가 생성 → 크리에이터 참여 | 선택 | 체험 후 공구 참여 가능, 체험 없이도 참여 가능 |
| 크리에이터픽 | 크리에이터가 카탈로그에서 추가 | 선택 | 체험 후 픽 추가 가능, 체험 없이도 추가 가능 |
| 제품 체험 | 크리에이터가 신청 → 브랜드 승인 | 핵심 | 공구/픽의 전 단계이자 독립적 플로우 |

**핵심**: 체험은 강제가 아님. 공구/픽을 하기 위해 반드시 체험할 필요 없음.
다만 체험을 거친 크리에이터의 공구 전환율이 높을 것이라는 가설.

## 7. 기존 온보딩 3단계와의 통합

이미 설계된 "체험 → 스토리 → 공구" 3단계 온보딩:

1. **Step 1 (체험 모드)** = 이 제품 체험 신청 기능
2. **Step 2 (스토리 테스트)** = 체험 후 스토리 링크로 반응 확인
3. **Step 3 (공구 도전)** = 데이터 기반 공구 참여 결정

제품 체험 신청은 이 3단계의 시작점이자 시스템적 기반.
